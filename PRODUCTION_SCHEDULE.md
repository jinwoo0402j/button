# 버튼 — v1 제품 제작 일정

기준일: 2026-07-12

제작 시작: 2026-07-13

목표 출시: **2026-07-17 금요일**

문제 발생 시 상한: **2026-07-21 화요일**

현재 기술 기준선은 `f6f5e39 feat: expand button game prototype`이다. 기존 윤리 질문, 룰렛, 피해자 정보, 돈, 목표, 효율 ×2, 인구, 저장과 31개 테스트를 재사용한다.

## 다시 계산한 원칙

> 기존 기능을 옮기거나 연결하는 일은 따로 하루씩 잡지 않는다. 작동하는 한 덩어리로 묶어 하루에 끝낸다.

- P0·P1·P2는 같은 화면과 상태를 재사용하므로 **하루**에 구현하고 검증한다.
- P3은 경제 계산과 연속 입력을 묶어 **하루**에 끝낸다.
- P4는 대량 사망과 엔딩을 묶어 **하루**에 끝낸다.
- P5는 저장과 전체 회귀를 묶어 **하루**에 끝낸다.
- 마지막 하루에는 전체 플레이, 수정, 승인과 출시만 한다.
- 검증 실패는 미리 일정에 넣지 않는다. 실제로 실패했을 때만 7월 20~21일을 사용한다.

## v1 범위

v1에는 다음만 포함한다.

1. 첫 윤리 질문부터 미완성 유혹까지의 첫 아크
2. 기하급수 경제와 죄책감 마모 뒤 연속 입력
3. 대량 사망과 인구 1명 엔딩
4. 저장, 회귀 검증, 승인 후 배포

확장 그래픽, 확장 스토리, 통계·기록·분석 시스템은 v1 뒤에 별도로 산정한다.

## P 완료마다 재귀 개선

P0, P1, P2가 같은 날 끝나더라도 각각 별도 완료 단위로 취급한다. 모든 P가 끝날 때 다음 다섯 항목을 사용자에게 제시한다.

1. 개발규칙
2. 개발목표
3. 일정
4. 기획
5. 구현

- `다음 P 진행`: 다음 작업을 시작한다.
- `현재 P 수정`: 수정하고 같은 재귀 개선을 다시 실행한다.
- `계획 재수립`: 다음 작업을 멈추고 범위와 일정을 다시 잡는다.

한 번의 재귀 개선은 3~5분 안에 읽을 수 있게 작성하고 별도 보고서 파일은 만들지 않는다.

## 하루당 업무량

### 7/13 월 — 첫 윤리 아크 전체

예상 집중 시간: **6~8시간**

오전에는 P0만 처리한다.

1. 연타해도 첫 룰렛에서는 한 명만 죽게 만든다.
2. `사람 → 죽음 → 돈 → 목표` 순서로 바꾼다.
3. 첫 피해자를 다음 입력 전까지 화면에 남긴다.
4. 첫 죽음의 업적과 축하를 없앤다.

중간에 처음 보는 사람 3명에게 P0만 시켜 본다.

- 통과: P0 재귀 개선을 제시하고 사용자가 진행을 선택하면 오후에 P1을 연결한다.
- 실패: P1·P2를 활성화하지 않고 P0만 수정한다.

통과했을 때의 오후 작업:

5. 대상이 끝나면 더 넓은 대상을 한 번 역제안한다.
6. P1 재귀 개선 뒤 사용자가 진행하면 잠긴 기능 하나와 미완료 업적 하나를 보여준다.
7. P2 재귀 개선을 실시한다.
8. 발견한 문제를 고치고 전체 테스트를 실행한다.

끝난 기준:

- 유저가 `내가 눌러서 죽었다`고 이해한다.
- 룰렛은 피해자를 골랐고 죽음과 돈은 고정이라고 이해한다.
- 첫 피해자가 돈보다 먼저 보이고 다음 선택까지 남는다.
- 계속과 종료가 모두 작동한다.
- 잠긴 요소는 각각 하나뿐이다.
- 자동 테스트가 통과한다.

### 7/14 화 — 경제와 연속 입력

예상 집중 시간: **5~7시간**

1. `DESIGN.md`에 확정된 여섯 단계 효율 사다리를 적용한다.
2. 다음 가격을 현재 1회 수익의 180회분으로 계산한다.
3. 죄책감 마모 전환 전에는 룰렛이 끝날 때까지 기다린다.
4. 전환 뒤에는 룰렛 중 다음 직접 입력을 받는다.
5. 클릭과 Space를 섞어도 입력·사망·돈이 정확히 맞게 한다.

끝난 기준:

- 가격과 효율 경계 테스트가 통과한다.
- 효율은 2억명에서 끝나고 다음 가격이 없다.
- 자동 클릭은 없다.
- 누른 횟수만큼만 사망과 돈이 발생한다.
- 연속 입력으로 중복 지급되지 않는다.
- P3 재귀 개선에서 사용자에게 다음 단계 진행 여부를 받는다.

### 7/15 수 — 대량 사망과 엔딩

예상 집중 시간: **5~7시간**

1. 한 입력으로 많은 사람이 죽는 계산을 만든다.
2. 대상군에 남은 사람보다 많이 죽이지 않게 한다.
3. 전 세계 단계에서는 `현재 인구 - 1`보다 많이 죽이지 않게 한다.
4. 대표 인물 한 명과 `그리고 N명이 더 죽었다`를 보여준다.
5. 지구에 한 명만 남으면 게임을 끝낸다.
6. 엔딩 뒤 버튼·돈·구매·인구 시계를 멈춘다.

끝난 기준:

- 대량 사망 수가 경계를 넘지 않는다.
- 인구가 정확히 1명에서 멈춘다.
- 엔딩 뒤 어떤 입력도 상태를 바꾸지 않는다.
- 조기 엔딩과 최종 엔딩이 모두 작동한다.
- P4 재귀 개선에서 사용자에게 다음 단계 진행 여부를 받는다.

### 7/16 목 — 저장과 전체 회귀

예상 집중 시간: **5~7시간**

1. 윤리 선택, 대상군, 돈, 효율과 엔딩을 함께 저장한다.
2. 기존 저장을 보존할지 초기화할지 확정한다.
3. 새로고침 중 거래가 사라지거나 두 번 지급되지 않게 한다.
4. 처음 질문부터 마지막 엔딩까지 전부 플레이한다.
5. 휴대폰, 데스크톱, 키보드와 모션 감소를 확인한다.
6. README와 실제 게임 설명을 맞춘다.

끝난 기준:

- 저장과 복원이 같은 상태를 만든다.
- 중복 사망·중복 지급이 없다.
- 모든 자동 테스트가 통과한다.
- 모든 주요 입력 방식으로 끝까지 플레이할 수 있다.
- P5 재귀 개선에서 사용자에게 출시 후보 진행 여부를 받는다.

### 7/17 금 — 수정, 승인, 출시

예상 집중 시간: **3~5시간**

1. 처음부터 끝까지 마지막으로 한 번 플레이한다.
2. 발견한 오류만 고친다.
3. 새 기능은 추가하지 않는다.
4. 사용자가 최종 화면을 확인한다.
5. 승인되면 커밋·푸시·배포한다.
6. 배포된 주소에서 한 번 더 확인한다.

끝난 기준:

- 출시 차단 오류가 없다.
- 사용자 승인이 있다.
- 배포본이 로컬과 같은 흐름으로 작동한다.

## 문제 발생 시에만 쓰는 일정

### 7/20 월 — 실패한 부분 수정

- 실패한 단계 하나만 고친다.
- 새 기능은 넣지 않는다.
- 해당 테스트와 전체 테스트를 다시 실행한다.

### 7/21 화 — 재검증과 출시

- 처음부터 끝까지 다시 플레이한다.
- 사용자 승인을 다시 받는다.
- 승인되면 출시한다.

## 왜 이 일정이 가능한가

- 새 화면을 거의 만들지 않고 기존 윤리 질문 영역을 재사용한다.
- 피해자, 돈, 목표와 룰렛이 이미 구현되어 있어 호출 순서만 바꾼다.
- 역제안은 대상군 하나와 전환 한 번만 만든다.
- 잠긴 기능과 업적은 기존 기능·목표 표현을 재사용한다.
- 정적 사이트라 백엔드와 서버 작업이 없다.
- v1에는 새 그래픽·스토리·통계 시스템이 없다.

## 일정 재산정 규칙

- 문구·표시 순서 문제: 같은 날 수정
- 한 기능의 상태 오류: **+1일**
- 저장 구조 또는 엔딩 구조 재작성: **+2일**
- 사용자 검증을 다시 해야 함: 참가자 준비 시간만큼 이동
- 그래픽·스토리·통계 시스템을 v1에 추가: 전체 일정 다시 계산

## 출시 차단 조건

- 입력과 죽음의 인과가 불명확함
- 피해자보다 돈·목표가 먼저 변함
- 초기 룰렛 중 중복 사망·중복 지급이 발생함
- 대상 범위가 수락 없이 넓어짐
- 효율·가격·잔액 계산이 깨짐
- 대상군 잔여 또는 `현재 인구 - 1`을 초과해 감소함
- 인구 1 이후 진행이 계속됨
- 저장 복원에서 상태가 사라지거나 두 번 정산됨
- 모바일·키보드·모션 감소 경로가 작동하지 않음
- 사용자 최종 승인이 없음
