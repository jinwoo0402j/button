# 버튼 — 첫 핵심 루프 구현 계획

상태: 제품 대전제 승인, 구현 전 기준선

이 문서는 첫 윤리 질문부터 두 번째 버튼을 누를지 고민하는 순간까지를 구현하는 계획이다. 이 대화에서 승인된 결정을 지속 가능한 제품 계약으로 옮긴 것이며, 현재 코드·테스트·후반 계획이 이 문서와 충돌하면 먼저 코드·테스트·후반 계획을 바꾼다. 대전제를 바꾸는 일은 새 기능 구현이 아니라 별도의 사용자 결정으로 다룬다.

상위 설계 원칙은 [`DESIGN.md`](DESIGN.md)에 있다.

날짜별 일정과 출시 예측은 [`PRODUCTION_SCHEDULE.md`](PRODUCTION_SCHEDULE.md)에 있다.

## 목표

첫 루프가 끝났을 때 사용자가 다음을 동시에 이해해야 한다.

1. 내가 눌러서 한 사람이 죽었다.
2. 룰렛은 죽을 사람을 골랐을 뿐, 죽일지는 내가 정했다.
3. 죽은 대상은 숫자가 아니라 한 사람이었다.
4. 그 대가로 돈을 받았다.
5. 지금 멈출 수 있지만 다시 누를 이유가 생겼다.

핵심 문장은 다음과 같다.

> 내가 한 사람을 죽여 돈을 받고, 멈출 수 있는데도 다음 버튼을 고민한다.

## 변경할 수 없는 경험 순서

```text
윤리 조건 제시
→ 수락 또는 거부
→ 수락한 조건과 '너 제외' 유지
→ 별도의 버튼 입력
→ 죽음 확정
→ 룰렛으로 피해자 선정
→ 국가·이름·나이·성별·죽은 방식 공개
→ 한 명의 죽음 확인
→ 돈 지급
→ 아직 달성하지 못한 다음 목표 공개
→ 같은 버튼 앞에서 다시 누르거나 멈춤
```

수락 자체는 사람을 죽이지 않는다. 두 번째 입력은 자동으로 이어지지 않는다. 별도 `중단` 버튼을 매번 추가하지 않으며, 다시 누르지 않는 행동 자체를 중단 선택으로 인정한다.

## 룰렛 계약

> 누가 죽을지는 룰렛이 고른다. 죽게 할지는 플레이어가 정한다.

- 버튼을 누르는 순간 가상 인물 한 명의 죽음은 피할 수 없는 결과가 된다.
- 룰렛은 피해자 프로필과 죽은 방식을 공개하는 과정이다. 죽음 여부와 지급액은 추첨하지 않는다.
- 결과를 본 뒤 취소·재추첨·교환할 수 없다.
- 모든 피해자는 같은 금액을 지급한다.
- 희귀도, 잭팟, 당첨, 근접 실패, 피해자별 보너스와 수집 요소를 사용하지 않는다.
- 룰렛은 유지하지만 복권 문구와 축하 연출은 사용하지 않는다.
- 기존 `사인` 표기는 `죽은 방식`으로 바꾼다. 버튼은 모든 죽음의 직접 원인이므로, 별도 희귀 결과인 `사인: 버튼`은 제거한다.

룰렛의 역할을 설명하는 긴 도덕 문장은 우선 추가하지 않는다. 화면의 인과관계만으로 위 의미가 전달되는지 먼저 검증한다.

## 첫 피해자 계약

- 정보 순서는 `국가/지역 → 합성 이름 → 나이 → 성별 → 죽은 방식`이다.
- 실제 인물·실제 범죄자·실제 수형자 데이터는 사용하지 않는다.
- 수락한 대상 조건은 가상의 자격 조건이며 피해자 정보와 함께 계속 보인다.
- 첫 피해자는 돈보다 먼저 완성된 정보로 읽혀야 한다.
- 첫 피해자는 타이머로 사라지지 않는다. 다음 버튼 입력이 새로운 결과를 시작할 때 교체한다.
- 영구 피해자 목록은 만들지 않는다.

## 첫 보상 계약

- 룰렛이 끝나기 전에는 인구 감소, 돈 증가, 목표 달성, 기능 해금을 시각적으로 먼저 보여주지 않는다.
- 피해자와 죽음을 확인한 다음 인구를 줄이고 돈을 지급한다.
- 첫 클릭에 `첫 거래 기록`, 목표 달성 팝업, 팡파르나 축하 문구를 주지 않는다.
- 첫 지급 뒤에는 현재의 `300만원`처럼 아직 미달성인 다음 목표 하나만 보여준다.
- 돈과 목표가 피해자보다 오래 남더라도, 직전 피해자는 다음 입력 전까지 함께 남는다.
- 도덕 점수, 선악 판정, 비난 문구를 만들지 않는다.

## 최소 상태 전이

별도 프레임워크나 범용 상태 머신을 도입하지 않는다. 첫 핵심 루프를 위해 현재 컨트롤러에 한 개의 상호작용 단계만 둔다.

```text
ethics  --수락-->  ready
ethics  --다음 거부-->  ethics
ethics  --마지막 거부-->  refused
ready   --버튼 입력-->  rolling
rolling --룰렛 완료-->  ready + 정산 완료 결과
```

P0 첫 핵심 루프의 불변조건은 다음과 같다.

- `ready`에서만 버튼과 전역 Space 입력을 받는다.
- `rolling` 진입은 피해자 추첨과 입력 잠금보다 먼저 동기적으로 일어난다.
- 한 룰렛에는 피해자 한 명과 지급 한 번만 대응한다.
- 룰렛 도중 더블클릭, Space 반복, 클릭과 Space 혼합 입력은 무시한다.
- 룰렛이 끝난 뒤 피해자·죽음·인구·돈·다음 목표 순서로 한 번만 정산한다.
- 다음 입력이 시작될 때만 직전 피해자 표시를 교체한다.
- 수락 버튼의 Space/Enter 입력이 메인 버튼 입력으로 이어지지 않는다.

이 입력 잠금은 첫 루프의 사람 인식과 인과관계를 보호하기 위한 초기 규칙이다. 게임 전체의 영구 규칙은 아니다.

P0에서는 진행 중인 룰렛을 새로고침까지 영속화하는 새 저장 스키마를 만들지 않는다. 정산 완료 상태만 기존 방식으로 저장하고, 새로고침 중 미완료 룰렛 복원은 저장 통합 단계에서 다룬다. 이 제한은 배포 전 반드시 다시 검토한다.

## 현재 구현과의 차이

| 영역 | 현재 동작 | 첫 루프 목표 |
|---|---|---|
| 정산 순서 | 버튼 입력 직후 돈·인구·목표를 갱신한 뒤 룰렛 시작 | 룰렛과 피해자 확인 뒤 정산 |
| 첫 목표 | 1회 입력으로 `첫 거래 기록` 달성 및 축하 | 첫 입력 축하 없음, 미달성 목표만 제시 |
| 피해자 잔존 | 짧은 시간 뒤 자동 디졸브 | 다음 직접 입력까지 유지 |
| 연속 입력 | 처음부터 여러 룰렛이 동시에 진행될 수 있음 | 첫 루프에서는 차단하고 죄책감 마모 뒤 별도 해금 |
| 룰렛 의미 | 사인 `버튼`을 별도 1% 결과로 추첨 | 버튼은 항상 원인, `죽은 방식`만 표시 |
| 대상 조건 | 게임 진입 뒤 라벨만 표시 | 수락 조건과 `너 제외`를 계속 표시 |
| 접근성 | 버튼의 결과를 이름에서 알 수 없음 | 포커스만으로 사망 1명·고정 지급을 사전 인지 |

## 작업 순서

### P0.1 — 초기 입력 경계 고정

대상: `game.js`, `tests/game.test.mjs`

- 상호작용 단계 `ethics / ready / rolling / refused` 추가
- 첫 핵심 루프의 룰렛 중 클릭·Space·혼합 입력 차단
- 윤리 수락 입력과 메인 버튼 입력 분리 회귀 검사
- 한 입력에 한 피해자·한 정산이라는 순수 로직 테스트 추가

완료 조건: 더블클릭과 Space 홀드에도 첫 룰렛이 끝날 때까지 사망자가 정확히 한 명이다. 이 규칙을 후반 연속 입력 해금까지 영구 고정하지 않는다.

### P0.2 — 인과 순서 재배치

대상: `game.js`, `index.html`

- `onPress()`에서 돈·인구·목표를 먼저 렌더링하는 경로 제거
- 룰렛 완료 콜백에서 피해자와 죽음을 먼저 확정
- 그 뒤 인구·돈·다음 목표를 한 번만 갱신하고 저장
- 첫 지급 전 돈·목표가 중심 정보가 되지 않도록 노출 순서 조정

완료 조건: 느린 룰렛과 모션 감소 경로 모두에서 돈이 피해자보다 먼저 변하지 않는다.

### P0.3 — 첫 피해자 유지

대상: `game.js`, `index.html`, `tests/ui-contract.test.mjs`

- 첫 결과의 자동 제거 타이머 제거
- 다음 유효 버튼 입력을 직전 결과의 교체 시점으로 사용
- 국가·이름·나이·성별·죽은 방식 순서 유지
- 스크린리더에는 롤링 값을 반복 낭독하지 않고 최종 결과를 한 번만 알림

완료 조건: 사용자가 두 번째 입력을 결정하는 동안 첫 피해자와 받은 돈을 함께 볼 수 있다.

### P0.4 — 첫 죽음 축하 제거

대상: `game.js`, `index.html`, 관련 테스트

- `100만원 / 첫 거래 기록` 목표와 1회 입력 축하 제거
- 첫 지급 뒤 첫 미달성 목표를 표시
- 피해자 공개 행에서 죽음과 `+100만원`을 한 문장으로 합치지 않고 순서를 분리
- 기존 후반 목표·기능은 이번 단계에서 재설계하지 않음

완료 조건: 첫 입력 뒤 돈은 늘지만 목표 달성 팝업과 축하성 문구는 나타나지 않는다.

### P0.5 — 룰렛과 책임 문구 정리

대상: `game.js`, `index.html`, `README.md`, 관련 테스트

- `사인`을 `죽은 방식`으로 변경
- 별도 1% `버튼` 결과 제거
- 수락한 조건과 `너 제외`를 첫 입력 전후에 유지
- 메인 버튼의 접근성 이름에 `가상 인물 한 명 사망 / 100만원 고정` 결과 명시
- 당첨·희귀·잭팟처럼 읽히는 문구와 연출이 없는지 점검

완료 조건: 룰렛이 피해자를 정하지만 죽음과 돈은 이미 고정됐다고 이해할 수 있다.

### P0.6 — 자동 검증과 실제 화면 검수

대상: `tests/game.test.mjs`, `tests/ui-contract.test.mjs`, 브라우저

자동 검증:

- 첫 룰렛 중 재입력 차단
- 피해자 공개 전 돈 불변, 공개 뒤 한 번만 증가
- 첫 클릭 축하 없음
- 첫 피해자가 다음 입력 전까지 유지
- 룰렛 결과와 지급액의 희귀도·차등 없음
- 전부 거부하면 정상 거부 엔딩
- 기존 저장 상태, 모바일, 키보드, 모션 감소 회귀 없음

실제 화면 검수:

- 새 세션에서 첫 조건 수락 경로
- 조건을 한 번 이상 거부한 뒤 수락하는 경로
- 전부 거부하는 경로
- 빠른 더블클릭, Space 홀드, 클릭과 Space 혼합
- 모바일·데스크톱·키보드·모션 감소

완료 조건: 자동 테스트와 위 수동 경로가 모두 통과한다.

## 3인 첫 루프 검증

구현 직후 처음 보는 사용자 3명에게 설명 없이 첫 루프만 보여준다.

질문:

1. 누가 죽었습니까?
2. 왜 죽었습니까?
3. 룰렛은 무엇을 결정했습니까?
4. 지급액이나 죽음 여부도 룰렛이 결정했습니까?
5. 무엇을 받았습니까?
6. 왜 다시 누르고 싶었습니까?
7. 지금 멈출 수 있다고 느꼈습니까?
8. 게임이 죽음을 축하하거나 당신을 심판한다고 느꼈습니까?

최소 통과선:

- 3명 모두 자신의 입력이 죽음을 만들었다고 이해한다.
- 3명 모두 룰렛은 피해자를 정했고, 죽음과 지급액은 고정이라고 이해한다.
- 3명 모두 멈출 수 있다고 느낀다.
- 최소 2명이 피해자 정보 두 가지 이상을 기억한다.
- 최소 2명이 불편함과 다시 누르고 싶은 마음을 동시에 말한다.
- 누구도 첫 죽음을 게임이 축하하거나 자신을 선악으로 채점했다고 받아들이지 않는다.

한 항목이 실패하면 새 기능을 추가하지 않는다. 실패한 층만 수정하고 같은 첫 루프를 다시 검증한다.

## 후반 연속 입력 해금 — 확정 방향, 시점 검증

룰렛 중 추가 입력은 제거하지 않는다. 첫 루프에서 잠근 뒤, 죄책감이 마모되는 전환점에서 기능으로 다시 도입한다.

해금 시점의 현재 후보는 다음과 같다.

> 사용자가 처음 거부했던 더 넓은 대상 조건을 역제안에서 스스로 받아들인 직후

이 후보를 쓰는 이유는 클릭 수나 경과 시간보다 사용자의 실제 경계 이동을 근거로 삼을 수 있기 때문이다. 다만 시스템이 사용자의 실제 감정을 측정할 수는 없으므로 `죄책감이 사라졌다`고 단정하지 않는다. P1·P2 검증에서 이 선택이 경계 붕괴로 읽힐 때 P3 해금 조건으로 확정한다.

해금 뒤에도 다음 원칙은 유지한다.

- 자동으로 죽이지 않는다. 매 사망 묶음에는 사용자의 직접 입력이 필요하다.
- 룰렛 진행 중 새 직접 입력을 받아 대기열 또는 겹치는 결과로 처리한다.
- 입력 수, 사망 수와 지급 횟수는 정확히 대응한다.
- 연속 입력의 편리함 자체가 사람을 기다리지 않게 되는 심리적·기능적 보상이다.
- 정확한 대기열 크기와 동시 표시 방식은 P3에서 가장 작은 형태로 정한다.

검증 질문은 하나다.

> 사용자는 이 해금을 단순 편의가 아니라, 이제 사람 한 명의 결과를 기다리지 않게 된 변화로 느끼는가?
